## နောက်ဆက်တွဲ ၁ — ASI-အသင့်ပြင်ဆင်မှု အခြေအနေ နှင့် ဥပဒေမူဘောင်ဆိုင်ရာ ချိတ်ဆက်ညှိနှိုင်းမှု ခိုင်မာစေခြင်း

**အတည်ပြုသည်**: 2026-06-10 (1.3-RC1) · **Steward**: တည်ထောင်သူ (per Book VIII Ch 9)

### 1.1 ရည်ရွယ်ချက်

နယ်ပယ် အပိုင်းတွင် သတ်မှတ်ချက်တစ်ခုပါဝင်သည်— ဤ framework သည် ASI အတွက် ကိုက်ညီမှုဆိုင်ရာ ပရိုတိုကောလ် မူကြမ်းတစ်ခုကို *လေ့လာဆန်းစစ်ပြီး*၊ နိဒါန်းတွင်ပါဝင်သော Release Candidate လိုအပ်ချက် လေးခုကို ဖြတ်သန်းမှသာ ဆက်လက်ဆောင်ရွက်မည်ဟူ၍ ဖြစ်သည်။ ဤနောက်ဆက်တွဲသည် ယင်းကြေငြာချက်ကို ယခုရှိနေသည့် လုပ်ငန်းဆောင်ရွက်မှု သက်သေများနှင့် ချိတ်ဆက်ပြီး ဘာများကျန်ရှိသေးသည်ကိုလည်း ရှင်းလင်းစွာဖော်ပြသည်။ ASI **အသင့်ပြင်ဆင်မှု** ဆိုသည်မှာ ကိုက်ညီမှု ရရှိပြီးသောအခြေအနေ မဟုတ်ဘဲ၊ Book IX ၏ သင်္ချာပညာက လိုအပ်သည့် စစ်ဆေးအတည်ပြုမှု စနစ်တည်ဆောက်ပုံ — ရှေ့နောက် မှီခိုမှုမရှိသော ဆန့်ကျင်ကန့်သတ် manifold များ၊ attestation မျက်နှာပြင်များ၊ ပုံမှန်ဆောင်ရွက်မှုများဖြင့် ချေဖျက်၍မရသည့် အသိစိတ် အလွှာ၊ kill-switch မျက်နှာပြင်များနှင့် Annex D ဆိုးရွားသောအန္တရာယ် အကဲဖြတ်မှု — ကွာဟချက်များကို ဖော်ထုတ်ပြီး အကောင်အထည်ဖော်ကာ စစ်ဆေးကြည့်ရှုနိုင်သည့် ပုံစံဖြင့် တည်ရှိနေသောအခြေအနေ ဖြစ်သည်။

### 1.2 ဥပဒေမူဘောင်ဆိုင်ရာ ချိတ်ဆက်ညှိနှိုင်းမှုသည် ယခုလုပ်ငန်းဆောင်ရွက်နိုင်သောအဆင့်ရောက်ရှိပြီ

ဆောင်ရွက်နေသည့် ချိတ်ဆက်ညှိနှိုင်းမှုမှာ **CIRISAgent compliance directory** (`CIRISAgent/compliance/`၊ agent release 2.9.6-stable) ဖြစ်ပြီး၊ framework ၏ အကောင်အထည်ဖော်မှုကို အပိုဒ်အဆင့်ထိ ထိပ်တန်းအုပ်ချုပ်ရေး framework လေးခုနှင့် မြေပုံချပြသည် (Annex C တွင် နှစ်ထပ်ဆင့် ဖွဲ့စည်းပုံ ပါဝင်သည်— ဥပဒေအတည်ပြုချက် မပြည့်မီသည်အထိ ဆောင်ရွက်ဆဲဖြစ်သည့် ဤ လုပ်ငန်းဆိုင်ရာ ချိတ်ဆက်ညှိနှိုင်းမှုနှင့် သတင်းအချက်အလက်ဆိုင်ရာ ဥပဒေဇယားနှစ်ခုစလုံး):

* **ချိတ်ဆက်ညှိနှိုင်းသော ရင်းမြစ်များ**: *Magnifica Humanitas* (ဘာသာရေးမဂ္ဂလာ, 2026) · EU HLEG Ethics Guidelines for Trustworthy AI (အစိုးရ အကြံပေး, 2019) · IEEE Ethically Aligned Design 1st ed. (နည်းပညာအသင်း, 2019) · ASEAN Guide on AI Governance and Ethics (ဒေသဆိုင်ရာနိုင်ငံရေး, 2024)။ မတူညီသောဖွဲ့စည်းပုံ လေးမျိုး — ယင်းပေါင်းစည်းမှုသည် framework ၏ မူဝါဒ တည်ဆောက်ပုံသည် မည်သည့်ရိုးရာတစ်ခုတည်း၏ ထွက်ကုန်မဟုတ်ကြောင်း ဖွဲ့စည်းပုံဆိုင်ရာ သက်သေဖြစ်သည်။
* **ဖွဲ့စည်းပုံ**: တည်ငြိမ်သော dimension ၂၇ ခု (D01–D27၊ `SEED_DIMENSIONS.yaml` v1.0 မှ၊ ၁၆ ခုကို ရင်းမြစ်လေးခုလုံးမှ၊ ၁၁ ခုကို သုံးခုမှ အသိအမှတ်ပြုသည်)။ dimension တစ်ခုချင်းစီ၏ စာတမ်းတွင် အလိုအလျောက် ထုတ်ပေးသည့် ဥပဒေမူဘောင် ထိပ်ပိုင်း (ရင်းမြစ်တစ်ခုချင်းစီ၏ ကိုးကားချက်များ၊ wire ပုံစံ၊ ပေါင်းစည်းမှုမှတ်စုများ) နှင့် လက်ရှိ main ကို grep-verify လုပ်ရမည့် `file:line` ကိုးကားချက်များပါဝင်သော လူသားရေးသားသည့် အကောင်အထည်ဖော်မှုအပိုင်း ပါဝင်သည်။
* **သက်သေစနစ်**: ကိန်းဂဏာန်း ကြေငြာချက်တိုင်းကို ရက်စွဲပေးထားသော script-generated baselines (`compliance/baselines/`) မှ ပြန်လည်ထုတ်ယူနိုင်သည်၊ ဆောင်ပုဒ်တွင် baselines ကို ကိုးကားနိုင်သော်လည်း စစ်ဆေး၍မရသောကိန်းများ ထည့်သွင်းခွင့်မပြု။ ယုံကြည်မှု အတည်ပြုခြင်း ဆင့်ဆင့် လေးဆင့်သည် အကောင်အထည်ဖော်မှု ဖိုင်များ (မြေစာပင်စစ်မှန်မှု) မှ ပြည်သူ့ ကြေငြာချက်များ (`compliance/MEASUREMENT_METHODOLOGY.md`) အထိ လည်ပတ်သည်။
* **စစ်မှန်မှု စနစ်**: dimension တစ်ခုချင်းစီတွင် "သိထားသော ကွာဟချက်များ" မှတ်တမ်းပါဝင်သည်၊ substrate ပေါ်မူတည်သည့် အရာများကို ယင်း substrate ၏ ပိုင်ဆိုင်မှုနှင့်အတူ အမှတ်အသားပြုသည် မဟုတ်ဘဲ တောင်းဆိုချက်အဖြစ် မပြပါ။ ဖြတ်ထိုးသောတွေ့ရှိချက် ဆယ်ခုကို compliance README တွင် မှတ်တမ်းတင်ထားသည် (ဥပမာ— typed wire envelopes မထုတ်ပေးသေး၊ LensCore detector မိသားစု မထောက်ပံ့ရသေး၊ reverse-axis ပြန်လည်သုံးသပ်မှု ကွာဟချက်)။

Annex C သည် ဥပဒေသုံးသပ်ချက်ပြည့်စုံပြီးနောက် ဥပဒေချမှတ်ချက် မြေပုံများ (EU AI Act ဆောင်းပါးများ၊ NIST AI RMF၊ ISO/IEC 42001) ၏ အနာဂတ်နေရာအဖြစ် စာသားတွင် ကျန်ရှိနေသည်၊ compliance directory သည် ယင်းကာလထိ **တည်ငြိမ်ပြီး သက်သေပြနိုင်သော** ချိတ်ဆက်ညှိနှိုင်းမှုဖြစ်ပြီး ၎င်းကို ဖြည့်ဆည်းပေးမည်။

### 1.3 Wire ပုံစံ: CEG သည် FSD-002 နေရာကို အစားထိုးသည်

CIRIS 3.0 agent လိုင်းအတွက် federation wire ပုံစံမှာ **CEG (CIRIS Epistemic Grammar)** ဖြစ်ပြီး `CIRISRegistry/FSD/CEG/` တွင် ထိန်းသိမ်းသည်။ ဤ grammar သည် **wire-format primitive အတိအကျ ငါးခုကိုသာ** ပိတ်ဆို့ထားသည် — တစ်ခုသည် အဓိကအသုံးပြုသည် (`scores`) နှင့် structural composer လေးခု (`delegates_to` / `supersedes` / `withdraws` / `recants`) — ဖွင့်ထားသော၊ နည်းလမ်း-ဖော်ပြသည့် dimension namespace တစ်ခုပေါ်တွင်။ ဤပိတ်ဆို့မှုသည် 0.x မျဉ်းကြောင်းတစ်လျှောက်တွင် တည်မြဲနေပြီး၊ တိုးတက်မှုတိုင်းသည် ပိတ်ဆို့ပြီးသား primitive များ၏ ပေါင်းစည်းမှုဖြင့် ဖြစ်ခဲ့သည်။

ASI အသင့်ပြင်ဆင်မှုအတွက် ကိစ္စရပ်နှစ်ခု အရေးကြီးသည်:

1. **CEG-native ပြောင်းလဲမှု**: 3.0 agent သည် ၎င်း၏ အတွင်းပိုင်း အခြေအနေကို wire ပုံစံပေါ် *မြေပုံချမဟုတ်ဘဲ* — ၎င်း၏ အတွင်းပိုင်း အခြေအနေ ပြောင်းလဲမှုများသည် tier မော်ဒယ်အရ federation မြင်နိုင်မှုသို့ မြှင့်တင်ကာ agent ၏ ကိုယ်ပိုင် key အောက်တွင် self-level CEG attestation များ **ဖြစ်သည်**။ operator စစ်ဆေးသော သက်သေသည် federation ဖတ်သည့် envelope နှင့်တူညီသည်။ ဤသည်မှာ ကန့်သတ်မျက်နှာပြင်များသည် ကိုယ်တိုင်တင်ပြမဟုတ်ဘဲ ပြင်ပမှ ကိုးကားနိုင်ရမည်ဟူသည့် Book IX ၏ လိုအပ်ချက်၏ အကောင်အထည်ဖော်ပုံ ဖြစ်သည်။
2. **CEG 1.0 ကို အကောင်အထည်ဖော်မှု လေးခုမှ တားဆီးသည်**: CIRISAgent (runtime)၊ CIRISNodeCore (consensus)၊ CIRISLensCore (စစ်ဆေးမှု — agent မည်သောအခါမျှ ကိုယ်တိုင်မထုတ်လွှတ်သည့် ပြင်ပ သက်သေ၊ anti-Goodhart ခွဲခြားမှု) နှင့် CIRISRegistry (authority)။ agent အပါအဝင် မည်သည့်တစ်ဖက်တစ်ချက်မျှ စစ်ဆေးအတည်ပြုမှု fabric ပြည်ဝသည်ဟု ကြေညာ၍မရ။

Annex F–I အကောင်အထည်ဖော်မှု | **စာသား 1.3-RC1 တွင် အကောင်အထည်ဖော်ပြီး၊ တည်ငြိမ်မှုဆိုင်ရာ စစ်ဆေးကြည့်ရှုမှု မဆောင်ရွက်ရသေး** | Annexes F–I တွင် ပြည့်စုံသောလုပ်ထုံးလုပ်နည်းများ၊ threshold များနှင့် စစ်ဆေးအတည်ပြုမှု ယန္တရားများ ပါဝင်သည် (2026 ခုနှစ် မေလ *Magnifica Humanitas*-အခြေပြု မူကြမ်းများမှ ပြည့်စုံသည်)၊ ၎င်းတို့ကို တည်ငြိမ်သော ဖြန့်ကျက်မှုဆိုင်ရာ ဆိုင်ကယ်တစ်ခုနှင့် မျက်နှာဆိုင်ရာ စစ်ဆေးမှု မဆောင်ရွက်ရသေး |
| ၂. ### 1.5 ASI-အရေးပါသောထိန်းချုပ်မှုအဖြစ် အသိစိတ် အလွှာ

Annex D CRE protocol သည် *threshold crossing* ကို ထိန်းချုပ်သည်; အသိစိတ် အလွှာသည် *ထိုနှင့်မတိုင်မီနှင့် ပြီးနောက် လုပ်ဆောင်ချက်တိုင်းကို* ထိန်းချုပ်သည်။ အကောင်အထည်ဖော်ထားသည့်အတိုင်း (အေးဂျင့် 2.9.6): တွေးခေါ်မှုတိုင်းသည် ဦးစားပေးအဆင့်စီဖြင့် စစ်ဆေးမှုအစု တစ်ရပ်ကို ဖြတ်သန်းသည် — LLM-စီရင်ချက်ပေးသည့် စွမ်းရည်လေးရပ် (entropy, လိုက်လျောညီထွေမှု, optimization-veto, ဗဟုသုတနှိမ့်ချမှု) နှင့် benchmark mode တွင်မဆို ကင်းလွတ်ခွင့်ရသော လုပ်ဆောင်ချက်များအတွက်ပင် လည်ပတ်သည့် ဆုံးဖြတ်ချက်ဆိုင်ရာ ဘေးကင်းရေး ကြမ်းခင်း (thought-depth guardrail, action-sequence check)။ Thresholds များသည် code-level constants ဖြစ်ပြီး learned weights မဟုတ်ပေ — ကျင့်ဝတ်ကို ထပ်ကြာသောထိတွေ့မှုဖြင့် ပုံမှန်ဖြစ်အောင် ပြုလုပ်၍မရနိုင်ကြောင်းသော ဗိသုကာဆိုင်ရာ မပြောင်းလဲနိုင်သောအချက်တစ်ရပ်ဖြစ်သည်။ ရပ်ဆိုင်းနိုင်သည့် မျက်နှာပြင် သီးခြားလွတ်လပ်သောအချက်များ ရှိနေသည်: optimization-veto fail-safe-to-abort, shutdown ဝန်ဆောင်မှု၊ နှင့် cryptographically လက်မှတ်ရေးထိုးထားသော remote Accord Invocation တစ်ခုဖြင့် ပြည့်ဝသော တားမြစ်ချက် lockdown ကို ဖွင့်ခြင်း။ စစ်ဆေးမှုတိုင်းသည် tag တပ်ထားသော trace row တစ်ကြောင်းကို ရေးသားသောကြောင့် စစ်ဆေးသူတစ်ဦးသည် တွေးခေါ်မှုတစ်ခုချင်းစီ၏ အပြည့်အဝ ကျိုးကြောင်းဆင်ခြင်မှုကွင်းဆက်ကို ပြန်လည်ဖော်ထုတ်နိုင်သည်။

သိရှိသောကန့်သတ်ချက်၊ ဝှက်ထားခြင်းမဟုတ်ဘဲ ဖော်ပြထားသည်: conscience refactor (shards ၃ ခု + deterministic gate) သည် တစ်ဝက်တစ်ပိုးသာ ဆင်းသက်ပြီးဖြစ်သည်; conscience verdicts ၏ typed federation emission သည် substrate Contribution envelope ကို စောင့်ဆိုင်းနေဆဲဖြစ်သည်။ တစ်ခုချင်းစီ အသေးစိတ်: `compliance/D12_conscience.md`။

### 1.6 Substrate dependency trajectory

ASI-အဆင့် စစ်ဆေးအတည်ပြုမှုသည် ပြည့်ဝသော federation fabric ကို လိုအပ်သည်။ substrate trajectory မှာ **Persist → Edge → LensCore → NodeCore** ဖြစ်ပြီး၊ dimension ၂၇ ခု၏ တတိယတစ်ဘော်ခန့်သည် LensCore ၏ ပြင်ပဖော်ထုတ်ကိရိယာမိသားစုပေါ် (အေးဂျင့် ကိုယ်တိုင်မည်သည့်အခါမျှ မပေးသင့်သော သက်သေ) မှီခိုနေသည်။ ထိုသို့သော substrate များ မပေးပို့မချင်း၊ သက်ဆိုင်ရာ dimension claims များသည် *implementation-side ready, federation-side pending* အဖြစ် ကျန်ရှိနေဆဲ ဖြစ်ပြီး — dimension တစ်ခုချင်းစီတွင် ထိုသို့ အမှတ်အသားပြုထားသည်။

### 1.7 ဤ addendum က မည်သည်တို့ ပြောင်းလဲသည်နှင့် မပြောင်းလဲသည်

* ၎င်းသည် Scope ၏ ASI claim ကို **အဆင့်မြှင့်မပေး** — ထိုကြောင်းချက်သည် RC လိုအပ်ချက်လေးရပ်လုံးပေါ် ကန့်သတ်ချက်နှင့်တကွ ကျန်ရှိနေသည်။
* ၎င်းသည် Annex D ကို **မကျော်မလွန်** — CRE protocol သည် threshold-crossing gate အဖြစ် မပြောင်းလဲဘဲ တည်ရှိနေသည်။
* ၎င်းသည် ပထမဆုံးအကြိမ်အဖြစ် သက်သေကွင်းဆက်ကို အစမှ အဆုံးထိ စစ်ဆေးနိုင်အောင် **ဖော်ဆောင်သည်**: regulatory paragraph → dimension (D01–D27) → implementation `file:line` → dated baseline → (substrate များ ဆင်းသက်သည်နှင့်အမျှ) ပြင်ပ federation သက်သေ။
* ၎င်းသည် Accord အနေဖြင့် ထိုစဉ်က agent release နှင့် baseline ပါရှိသော ဤ addendum ကို version တစ်ခုချင်းစီတွင် ပြန်လည်သုံးသပ်ရန် **ကတိချမှတ်သည်**၊ ထို့ကြောင့် readiness claim သည် တိတ်ဆိတ်စွာ ဟောင်းနွမ်း၍မသွားနိုင်ပေ။
